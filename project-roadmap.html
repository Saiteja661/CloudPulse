01
What You're Building

CloudPulse is an uptime monitoring platform where users register URLs, and the system checks them every minute, stores metrics, triggers alerts on downtime, and exposes a live dashboard. It's a complete SaaS product skeleton — with a Python Flask API backend, a React/HTML frontend, a full DevOps pipeline, and deep AWS integration.

8-Week Build Timeline
1
WEEK 1 Infrastructure Foundation with Terraform IaC AWS
Set up all AWS infrastructure as code using Terraform before writing a single line of application code. This is how real SRE teams work.
  • 1.1
    Create a GitHub repo cloudpulse with directories: /terraform, /backend, /frontend, /lambda, .github/workflows
    Git
  • 1.2
    Write Terraform modules for: VPC with public + private subnets, Internet Gateway, NAT Gateway (avoid NAT — use VPC Endpoints instead to stay free), Security Groups, and IAM roles. terraform init && terraform plan -out=tfplan && terraform apply tfplan
    Terraform
  • 1.3
    Provision EC2 t2.micro in public subnet with a security group that allows only port 443/80/22. Use key pair for SSH. Add user-data script to install Python, pip, and nginx on boot.
    EC2
  • 1.4
    Provision RDS MySQL db.t3.micro in private subnet. Store credentials in AWS Secrets Manager (free tier). Configure security group to only allow EC2's security group as a source.
    RDSIAM
  • 1.5
    Create S3 bucket for frontend static hosting + another bucket for Terraform remote state with DynamoDB table for state locking. Enable versioning on both. aws s3api put-bucket-versioning --bucket cloudpulse-tf-state --versioning-configuration Status=Enabled
    S3
  • 1.6
    Configure Route 53 hosted zone for saiteja.shop. Create A record for root and CNAME for api.saiteja.shop pointing to EC2 elastic IP. Request ACM SSL certificate for both.
    Route 53ACM
2
WEEK 2 Flask Backend API + Database Schema Python EC2
Build the core REST API that handles URL registration, check history retrieval, and alerting. Real microservice patterns — not a toy app.
  • 2.1
    Design MySQL schema: tables for users, monitors (url, name, interval, status), check_results (monitor_id, status_code, latency_ms, is_up, checked_at), incidents (start_time, end_time, duration).
    MySQL
  • 2.2
    Build Flask API with endpoints: POST /monitors, GET /monitors, GET /monitors/:id/history, GET /stats, GET /health. The /health endpoint is critical for CloudWatch.
    Flask
  • 2.3
    Containerize with Docker. Write a production-grade Dockerfile using a slim Python base image. Run Gunicorn (not Flask dev server) as the WSGI server with 4 workers. docker build -t cloudpulse-api . && docker run -p 5000:5000 cloudpulse-api
    Docker
  • 2.4
    Set up Nginx as a reverse proxy on EC2, terminating SSL with the ACM cert. Nginx forwards traffic to Gunicorn on 127.0.0.1:5000. Write a systemd service file so the app restarts on crash.
    EC2
  • 2.5
    Write unit tests using pytest for each endpoint. Write an integration test that spins up a local SQLite DB and validates the full create → query → delete flow. pytest tests/ -v --cov=app --cov-report=html
    Testing
3
WEEK 3 Lambda Polling Engine + EventBridge Lambda EventBridge
The heart of CloudPulse — a serverless function that wakes up every minute, checks all registered URLs, writes results to the database, and triggers alerts on failures.
  • 3.1
    Write the Lambda function in Python. It queries the Flask API for all active monitors, makes HTTP requests to each URL (with a 10s timeout), and posts results back to POST /internal/check-result.
    Lambda
  • 3.2
    Deploy the Lambda inside the same VPC so it can reach the RDS directly. Package it using a Lambda Layer for dependencies (requests, boto3). Set memory to 128MB to stay free. zip -r lambda.zip lambda_function.py && aws lambda update-function-code --function-name cloudpulse-poller --zip-file fileb://lambda.zip
    Lambda
  • 3.3
    Create an EventBridge rule with a cron expression rate(1 minute) that triggers the Lambda. AWS free tier gives you 1M Lambda invocations/month — checking 10 URLs = 14,400 invocations/day, well within limits.
    EventBridge
  • 3.4
    Implement downtime detection: if a URL fails 2 consecutive checks, create an incident record and publish a message to an SNS topic. SNS sends an email alert to you instantly.
    SNSIAM
4
WEEK 4 CI/CD Pipeline with GitHub Actions GitHub Actions CI/CD
The pipeline should be as close to what you built at Exalttek as possible — security scanning, test gates, automated deploy, one-click rollback.
  • 4.1
    Write the CI workflow (ci.yml): triggers on PR to main. Jobs: (1) lint with flake8 + black, (2) test with pytest + coverage gate (fail if <80%), (3) security with bandit (Python SAST) and safety (dependency CVE check).
    CI
  • 4.2
    Write the CD workflow (deploy.yml): triggers on push to main after CI passes. Build Docker image, push to ECR (free tier: 500MB/month), SSH into EC2, pull new image, restart the container. docker build -t $ECR_REGISTRY/cloudpulse-api:$SHA . && docker push $ECR_REGISTRY/cloudpulse-api:$SHA
    ECRCD
  • 4.3
    Implement rollback: tag the previous image as last-stable before every deploy. Add a manual rollback.yml workflow triggered via GitHub UI that redeploys the last-stable image.
    Rollback
  • 4.4
    Store all secrets (DB password, AWS keys, EC2 SSH key) in GitHub Actions Secrets. Use OIDC (OpenID Connect) instead of static AWS keys — add an IAM role that trusts GitHub Actions OIDC provider. This is production-grade security.
    OIDCIAM
  • 4.5
    Write a Terraform CD sub-workflow: on changes to /terraform/**, run terraform plan in CI and post the plan as a PR comment. On merge, run terraform apply automatically. This is GitOps.
    GitOpsTerraform
5
WEEK 5 Observability — CloudWatch, Prometheus & Grafana Monitoring Alerts
Replicate the observability stack from your resume — custom dashboards, alert policies, and log aggregation. No Datadog needed; everything here is free.
  • 5.1
    Install the CloudWatch Agent on EC2. Configure it to ship custom metrics: API request count, response time p50/p95/p99, error rate, active monitors count. Define metric namespaces.
    CloudWatch
  • 5.2
    Create 12 CloudWatch Alarms: CPU >70%, memory >80%, API error rate >5%, RDS connections >50, Lambda errors >0, Lambda duration >5s, etc. Route alarm actions to SNS → email.
    Alarms
  • 5.3
    Add a /metrics endpoint to the Flask API that exposes Prometheus-format metrics using the prometheus-client Python library. Track request count, latency histograms, and error counts. pip install prometheus-client && flask route → /metrics
    Prometheus
  • 5.4
    Install Prometheus + Grafana on the EC2 (Docker Compose for simplicity). Prometheus scrapes localhost:5000/metrics every 15s. Grafana connects to Prometheus as data source.
    Grafana
  • 5.5
    Build 10 Grafana dashboards: API Overview (RPS, latency, errors), URL Uptime % per monitor, Incident History timeline, EC2 resource utilization, Lambda invocations & duration, Database connection pool, and more.
    Dashboards
  • 5.6
    Configure CloudWatch Log Groups with structured JSON logging from Flask (using Python's python-json-logger). Set log retention to 7 days to stay within free tier. Create log metric filters for ERROR-level events.
    Logs
6
WEEK 6 Frontend Dashboard + CDN Deployment CloudFront S3
Build a live status dashboard that calls your API and visualizes uptime history, response times, and incidents — deployed globally via CloudFront.
  • 6.1
    Build a clean HTML/CSS/JS (or React) status dashboard with: live uptime percentages per URL, response time charts (Chart.js), incident history list, and a global "All Systems Go" / "Incident Detected" banner.
    Frontend
  • 6.2
    Configure the S3 bucket for static website hosting. Set bucket policy for public read. Enable S3 server access logging to a separate log bucket. aws s3 sync ./frontend/dist s3://cloudpulse-frontend --delete
    S3
  • 6.3
    Create a CloudFront distribution with S3 as origin. Enable HTTPS-only, attach your ACM certificate. Set up custom error page for 404 → index.html (for SPA routing). Enable CloudFront access logs.
    CloudFront
  • 6.4
    Add frontend deployment to your GitHub Actions CD pipeline: build step → npm run build → sync to S3 → invalidate CloudFront cache. The whole frontend deploy becomes one-click. aws cloudfront create-invalidation --distribution-id $CF_ID --paths "/*"
    CD
7
WEEK 7 Security Hardening + IAM Audit Security IAM
Mirror the IAM security review you did at Exalttek. Audit every role, apply least-privilege, enable GuardDuty, and document the findings like a real cloud security report.
  • 7.1
    Audit all IAM roles and policies. Use IAM Access Analyzer to find over-privilege. Identify any role with wildcard (*) actions or resources and replace with specific permissions. Document findings in a markdown report.
    IAM
  • 7.2
    Enable AWS GuardDuty (30-day free trial) for threat detection. Enable AWS Config to track resource configuration changes. Set up an SNS alert for any config change to EC2 security groups.
    GuardDuty
  • 7.3
    Move all secrets (DB password, API keys) to AWS Secrets Manager. Update the Flask app and Lambda to fetch secrets at runtime using boto3. Rotate the DB password manually and confirm the app auto-picks it up.
    Secrets
  • 7.4
    Add rate limiting to the Flask API using Flask-Limiter (e.g., 60 req/min per IP). Add input validation + parameterized SQL queries to prevent injection. Run OWASP ZAP or Nikto against the API and document findings.
    AppSec
  • 7.5
    Enable S3 bucket versioning and object lock on the Terraform state bucket. Enable S3 bucket encryption (SSE-S3). Block all public access on non-frontend buckets. Enable CloudTrail for full API audit logging.
    CloudTrailS3
8
WEEK 8 Kubernetes Local Cluster + Documentation K8s Helm
EKS is not free-tier eligible, so we simulate it locally with kind or minikube, deploy the app using Helm charts, and document the EKS-equivalent path for your resume.
  • 8.1
    Install kind (Kubernetes in Docker) locally. Create a 3-node cluster config. Push your Docker image to a local registry. kind create cluster --config kind-config.yaml && kubectl cluster-info
    Kubernetes
  • 8.2
    Write Helm charts for the API: Deployment, Service, ConfigMap, HorizontalPodAutoscaler (HPA). Use values.yaml to parameterize env per environment (dev/prod). Deploy to kind. helm install cloudpulse-api ./charts/api --values ./charts/api/values.prod.yaml
    Helm
  • 8.3
    Configure HPA to scale from 1→3 replicas based on CPU >60%. Use kubectl run --generator to simulate load and watch pods scale horizontally. Screenshot and document for your resume.
    HPA
  • 8.4
    Write a comprehensive README.md with: architecture diagram (ASCII or draw.io), local setup guide, deployment runbook, observability docs, and a "lessons learned" section. Write a dev.to or Medium article about this project.
    Docs
  • 8.5
    Final polish: record a 5-minute Loom demo video of the live system. Add the GitHub link to your resume and LinkedIn. Tag it with: AWS, Terraform, GitHub Actions, Docker, Kubernetes, Prometheus, Grafana, Python.
    Portfolio
